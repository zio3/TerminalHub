# リリース

TerminalHubの新バージョンをリリースする。以下の手順を順番に実行すること。

**重要**: 手順2の「リリースノート案の提示と合意」でユーザーの承認を得るまで、バージョン更新以降（コミット・プッシュ・タグ・GitHubリリース・Discord投稿）には進まないこと。これらは不可逆な作業のため、必ず合意を取ってから実行する。

## 手順

### 1. 変更内容の確認
- `git log --oneline` で前回タグからのコミットを確認
- リリースノートに含める変更内容を整理
- 次バージョン番号を決める（前回バージョン+1）

### 2. リリースノート案の提示と合意（★確認ゲート）
以下をユーザーに提示し、**明示的な合意を得てから次に進む**：
- **次バージョン番号**（例: 1.0.62）
- **リリースノート案**（下記フォーマット。ユーザー向けに主要な変更を簡潔に）
```markdown
## 新機能
- 機能の説明

## 改善
- 改善の説明

## バグ修正
- 修正の説明
```
該当するセクションのみ記載する。

このタイミングで:
- ユーザーが「内部的な話は省略」「この項目は外して」等の調整を求めたら反映して再提示する。
- バージョン番号の妥当性も確認する。
- **合意（「OK」「これで進めて」等）が得られるまで、手順3以降は実行しない。** 合意が曖昧なら確認する。

### 3. バージョン更新
合意後、以下の2ファイルのバージョンを更新する：
- `TerminalHub/TerminalHub.csproj` の `<Version>`
- `installer/TerminalHub.iss` の `#define MyAppVersion`

### 4. コミット・プッシュ
- バージョン更新をコミット: `chore: バージョンをX.X.Xに更新`
- `git push origin master`

### 5. GitHubリリース作成
`gh release create` で `--notes` に手順2で合意したリリースノートを記載する。

### 6. Discordお知らせ投稿
TerminalHubサーバーの「お知らせ」チャンネル（ID: `1488867349246513264`）にリリース案内を投稿する。
- `mcp__discord__discord_execute` の `messages.send` を使用
- ユーザー向けに主要なメリットを簡潔に伝える（技術的な詳細は不要）
- GitHubリリースページへのリンクを含める
